# Prompt — Discagem Preditiva na Página de Leads (Cursor AI / Claude)

---

## Contexto do Projeto

Estou desenvolvendo um sistema de callcenter com Asterisk e WebRTC. A aplicação já possui integração VoIP funcionando no navegador via WebRTC. Preciso implementar **discagem preditiva** na página `Leads`, onde múltiplas operadoras podem estar ativas simultaneamente, cada uma realizando 4 chamadas simultâneas.

**Stack:**
- Frontend: React
- Backend: Node.js
- Banco de dados: PostgreSQL (Supabase)
- Asterisk: comunicação via AMI (porta 5038)


**Regra de disponibilidade de um lead para discagem:**
```sql
WHERE leads_status = 'Nunca Ligado'
  AND operator_code_id IS NULL
```

> Um lead com `operator_code_id IS NOT NULL` está reservado por uma operadora e não deve ser selecionado por nenhuma outra.

## Regras de negócio

### Discagem preditiva por operadora

- Cada operadora ativa realiza **exatamente 4 chamadas simultâneas** por rodada.
- Se houver 5 operadoras ativas ao mesmo tempo, o sistema terá até **20 chamadas em andamento** (5 × 4).
- As 4 chamadas de cada operadora são disparadas ao mesmo tempo, para 4 leads diferentes.
- Assim que **qualquer uma das 4 chamadas for atendida**, as outras 3 são encerradas via AMI (`Action: Hangup`).
- Após encerrar o grupo (alguém atendeu OU todas caíram), o sistema dispara automaticamente o próximo grupo de 4 para aquela operadora (loop contínuo enquanto ativa).

### Seleção de telefone para discar

Cada lead pode ter até 6 telefones (`leads_tel_1` a `leads_tel_6`). A discagem deve usar o **primeiro telefone não nulo** disponível. Sugestão de prioridade: `leads_tel_1 → leads_tel_2 → ... → leads_tel_6`. Se um lead não tiver nenhum telefone válido, deve ser ignorado na seleção.

## Atualizações no banco de dados

### Ao INICIAR o grupo de 4 chamadas:
```sql
UPDATE leads
SET operator_code_id = $operator_code_id,
    leads_status = 'Em Chamada'
WHERE leads_id = ANY($leads_ids);  -- array com os 4 IDs
```

### Quando ALGUÉM ATENDER (e as demais forem canceladas):
```sql
-- Apenas para os leads cujas chamadas foram CANCELADAS (não o que atendeu):
UPDATE leads
SET operator_code_id = NULL,
    leads_status = 'Nunca Ligado'
WHERE leads_id = ANY($cancelled_leads_ids);  -- array com os 3 IDs que não atenderam

-- O lead que atendeu NÃO deve ser alterado aqui — segue o fluxo normal da chamada.
```

### Quando NINGUÉM ATENDER (todas as 4 caírem):
```sql
UPDATE leads
SET operator_code_id = NULL,
    leads_status = 'Nunca Ligado'
WHERE leads_id = ANY($all_leads_ids);  -- array com os 4 IDs
```

## O que precisa ser implementado

### 1. Backend — Serviço AMI (Node.js)

Crie ou adapte o serviço de conexão com o Asterisk AMI:

```js
// backend/services/amiService.js
// Responsabilidades:
// - Manter conexão persistente com o AMI (reconexão automática em caso de falha)
// - Expor método: originateGroup(operator, leads) → dispara 4 chamadas Async
// - Monitorar eventos: Hangup, OriginateResponse
// - Mapear ActionID → { lead_id, operator_code_id, channel }
// - Emitir eventos internos quando um grupo for resolvido (answered | no-answer)
```

**Parâmetros do `Action: Originate`:**
```
Action: Originate
Channel: SIP/<numero_do_lead>
Context: predictive-dialing        ← contexto no extensions.conf do Asterisk
Exten: s
Priority: 1
Timeout: 30000                     ← 30 segundos
ActionID: <uuid-unico>             ← para rastrear qual lead corresponde
Async: true
Variable: GROUP_ID=<group_uuid>,OPERATOR=<operator_code_id>,LEAD_ID=<lead_id>
```

**Lógica de eventos AMI a monitorar:**
- `OriginateResponse` com `Response: Success` → chamada conectada (alguém atendeu)
  - Encerrar os outros 3 canais via `Action: Hangup`
  - Atualizar BD: reverter os 3 cancelados para `'Nunca Ligado'` / `operator_code_id = NULL`
- `Hangup` para todos os 4 canais sem nenhum `OriginateResponse: Success` → ninguém atendeu
  - Atualizar BD: reverter os 4 leads
- Em ambos os casos: emitir evento via WebSocket para o frontend e disparar próximo grupo

### 2. Backend — Endpoint de discagem (Node.js)

```
POST /api/leads/predictive-dial/start
Body: { operator_id, operator_code_id }

POST /api/leads/predictive-dial/stop
Body: { operator_id }
```

**Lógica do endpoint `start`:**
1. Buscar 4 leads disponíveis no PostgreSQL com lock:
```sql
SELECT leads_id, leads_name,
       COALESCE(leads_tel_1, leads_tel_2, leads_tel_3,
                leads_tel_4, leads_tel_5, leads_tel_6) AS phone
FROM leads
WHERE leads_status = 'Nunca Ligado'
  AND operator_code_id IS NULL
  AND COALESCE(leads_tel_1, leads_tel_2, leads_tel_3,
               leads_tel_4, leads_tel_5, leads_tel_6) IS NOT NULL
ORDER BY leads_id
LIMIT 4
FOR UPDATE SKIP LOCKED;   ← evita que duas operadoras peguem os mesmos leads
```
2. Atualizar os 4 leads (`operator_code_id`, `leads_status = 'Em Chamada'`) na mesma transação.
3. Disparar as 4 chamadas via `amiService.originateGroup()`.
4. Retornar os leads selecionados ao frontend via WebSocket.

### 3. Backend — WebSocket (Node.js)

Emitir os seguintes eventos para o frontend:

```js
// Grupo iniciado
socket.emit('dialing:group-started', {
  operator_id,
  group_id,
  leads: [{ leads_id, leads_name, phone }]  // 4 leads
});

// Chamada atendida
socket.emit('dialing:answered', {
  operator_id,
  group_id,
  answered_lead_id,
  cancelled_lead_ids: [id1, id2, id3]
});

// Nenhuma chamada atendida
socket.emit('dialing:no-answer', {
  operator_id,
  group_id,
  lead_ids: [id1, id2, id3, id4]
});

// Próximo grupo iniciado automaticamente (loop)
socket.emit('dialing:next-group', { operator_id });

// Sem mais leads disponíveis
socket.emit('dialing:exhausted', { operator_id });
```

### 4. Frontend — Página Leads (React)

Na página `Leads.jsx`, implemente:

**Controles de discagem:**
- Um botão **"Iniciar Discagem"** / **"Parar Discagem"** por operadora (ou global)
- Indicador de status: "Discando..." / "Parado" / "Sem leads disponíveis"

**Painel de chamadas em andamento:**
- Exibir em tempo real os 4 leads do grupo atual por operadora
- Status visual de cada lead: `Em Chamada` (animação), `Atendido` (verde), `Sem Resposta` (cinza)
- Contador: "X chamadas em andamento | Y grupos completados"

**Atualização da listagem de leads:**
- Ao receber `dialing:group-started`: marcar os 4 leads como `Em Chamada` na tabela
- Ao receber `dialing:answered`: reverter os 3 cancelados para `Nunca Ligado` na tabela
- Ao receber `dialing:no-answer`: reverter os 4 leads na tabela
- Usar `invalidateQueries` (React Query) ou atualização otimista de estado local

## Fluxo completo esperado

```
Operadora A clica em "Iniciar Discagem"
  ↓
POST /api/leads/predictive-dial/start { operator_id: A, operator_code_id: 42 }
  ↓
Backend: SELECT 4 leads com FOR UPDATE SKIP LOCKED
Backend: UPDATE leads SET operator_code_id=42, leads_status='Em Chamada' WHERE leads_id IN (1,2,3,4)
Backend: AMI origina 4 chamadas Async (ActionID mapeado para cada lead_id)
Frontend recebe: dialing:group-started → atualiza tabela
  ↓
  ├── Lead 2 atende (OriginateResponse: Success para ActionID do lead 2)
  │     ↓
  │   AMI: Hangup nos canais dos leads 1, 3, 4
  │   BD: UPDATE leads SET operator_code_id=NULL, leads_status='Nunca Ligado' WHERE leads_id IN (1,3,4)
  │   Frontend recebe: dialing:answered { answered: 2, cancelled: [1,3,4] }
  │   → Loop: backend busca próximos 4 leads e repete
  │
  └── Ninguém atende (todos os 4 Hangup sem OriginateResponse: Success)
        ↓
      BD: UPDATE leads SET operator_code_id=NULL, leads_status='Nunca Ligado' WHERE leads_id IN (1,2,3,4)
      Frontend recebe: dialing:no-answer { leads: [1,2,3,4] }
      → Loop: backend busca próximos 4 leads e repete

Operadora A clica em "Parar Discagem"
  ↓
POST /api/leads/predictive-dial/stop { operator_id: A }
Backend: encerra o ciclo (não dispara próximo grupo) e faz hangup nas chamadas ativas
```

## Controle de concorrência entre operadoras

- O `FOR UPDATE SKIP LOCKED` no PostgreSQL garante que duas operadoras nunca selecionem os mesmos leads.
- `operator_code_id IS NOT NULL` funciona como "reserva" — qualquer lead com esse campo preenchido está bloqueado para outras operadoras.
- O estado de quais operadoras estão ativas e seus `group_id` ativos deve ser mantido em memória no backend (Map ou objeto simples), não no banco.

---

## Restrições importantes

- **Não alterar** `leads_status` do lead que atendeu — apenas os cancelados.
- O loop para automaticamente quando:
  - A operadora clicar em "Parar"
  - Não houver mais leads com `leads_status = 'Nunca Ligado'` e `operator_code_id IS NULL`
- Tratar reconexão automática do AMI em caso de queda.
- Logar cada grupo: quais `leads_id`, qual `operator_code_id`, quem atendeu, horário — para auditoria.
- Garantir que ao reiniciar o servidor, leads presos com `leads_status = 'Em Chamada'` sejam revertidos para `'Nunca Ligado'` (cleanup no startup).
